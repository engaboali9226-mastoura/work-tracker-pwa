# ADR-014 Entity Key Strategy

## Status

Accepted

## Date

2026-06-23

---

## Context

The system contains multiple domain entities that require unique identifiers.

Examples include:

* Area
* Project
* Site
* WorkDay
* Task
* TaskCategory
* Note
* EventLog

A consistent identifier strategy is required to ensure:

* Global uniqueness
* Simplicity
* Future storage compatibility
* Support for offline creation scenarios

---

## Decision

All entity identifiers shall use UUID v4 strings.

Identifiers shall be generated by the application using:

```ts
crypto.randomUUID()
```

No sequential identifiers or centralized counters shall be used.

---

## Applies To

The following identifiers shall use UUID v4:

* areaId
* projectId
* siteId
* categoryId
* workDayKey
* taskKey
* noteId
* eventId

Future entities shall follow the same strategy unless explicitly documented otherwise.

---

## Rationale

UUID v4 provides:

* Globally unique identifiers
* No dependency on centralized ID generation
* Compatibility with browser environments
* Compatibility with future storage providers
* Support for offline entity creation

The browser-native implementation avoids introducing additional dependencies.

---

## Consequences

### Advantages

* Globally unique identifiers
* Offline-friendly generation
* No coordination required between services
* Compatible with future persistence layers
* No additional libraries required

### Disadvantages

* Identifiers are longer than sequential keys
* Less human-readable during manual inspection
