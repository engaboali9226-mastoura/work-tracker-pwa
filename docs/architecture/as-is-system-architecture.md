# Work Tracker V2

# Current System Architecture (As-Is)

**Document Status:** Draft

**Architecture Style:** Domain-Centric Architecture

**Purpose**

This document describes the current implementation of the Work Tracker V2 system as it exists today. The document is based on the implemented source code rather than future plans or intended designs.

---

# 1. System Overview

## Purpose

Work Tracker V2 is a business process management system designed to organize and monitor daily work execution.

Instead of focusing on simple CRUD operations, the system models the real business workflow through business services, domain models, and business rules.

The system currently supports:

- Work Day Management
- Task Lifecycle Management
- Production Record Management
- Event Logging
- Classification Management
- Reference Data Management

---

# 2. Architectural Goals

The current implementation is designed to achieve the following goals:

- Separate business logic from infrastructure.
- Keep business rules independent from storage technologies.
- Organize code around business domains.
- Support multiple persistence providers.
- Make business services reusable.
- Centralize dependency composition.
- Allow future expansion without modifying existing business logic.

---

# 3. Architectural Principles

The current implementation follows several architectural principles.

## Domain-Centric Design

Business capabilities are organized into independent domains.

Each domain owns its own:

- Models
- Repository Interfaces
- Business Services
- Business Rules
- Domain Errors

---

## Use Case Driven Services

Services represent business operations instead of generic CRUD operations.

Examples include:

- Create Task
- Create Work Day
- Change Task State
- Record Production
- Query Tasks
- Query Work Days

Each service performs a complete business operation.

---

## Repository Pattern

Business services never communicate directly with storage.

Instead they depend on repository interfaces.

The infrastructure layer provides concrete implementations for these repositories.

---

## Dependency Injection

All repositories and business services are composed inside a single Application Container.

Business services never create their own dependencies.

Dependencies are injected during application startup.

---

## Separation of Concerns

Responsibilities are divided into specialized components.

Business logic does not contain infrastructure concerns.

Infrastructure does not contain business rules.

Configuration is isolated from business logic.

# 6. Application Composition Component

## Responsibility

The Application Composition component is responsible for constructing the entire application.

It acts as the Composition Root of the system by creating repositories, creating business services, and wiring all dependencies together before the application becomes available to the user.

No business logic exists inside this component.

Its only responsibility is object composition and dependency management.

---

## Inputs

The component requires:

- Environment Configuration
- Repository Provider Selection

The environment configuration determines which repository implementation should be used during runtime.

---

## Internal Processing

The initialization sequence follows these steps.

### Step 1

Load application configuration.

The configuration includes the active repository provider and infrastructure settings.

↓

### Step 2

Create the Repository Factory.

The factory decides which repository implementation will be used.

↓

### Step 3

Create repository instances.

Repositories are created only once during application startup.

Current repositories include:

- Task Repository
- Work Day Repository
- Event Repository
- Area Repository
- Site Repository
- Project Repository
- Contractor Repository
- Task Category Repository
- Sub Category Repository
- Production Record Repository

↓

### Step 4

Create business services.

Each service receives only the dependencies required to perform its business responsibility.

Current services include:

- Task Creation Service
- Task Lifecycle Service
- Task Query Service

- Work Day Creation Service
- Work Day Lifecycle Service
- Work Day Query Service

- Production Record Creation Service
- Production Record Query Service

- Area Query Service
- Site Query Service
- Project Query Service
- Contractor Query Service

- Task Category Query Service
- Sub Category Query Service

↓

### Step 5

Expose a fully initialized Application Container.

The remainder of the application interacts only with this container.

---

## Outputs

The component produces a fully initialized object graph containing:

- Repository instances
- Business services
- Shared dependencies

No additional runtime composition is required.

---

## Dependencies

Application Composition depends on:

- Environment Configuration
- Repository Factory
- Infrastructure Layer

It does not depend on any user interface component.

---

## Business Rules

The component does not contain business rules.

It is responsible only for dependency composition.

---

## Current Implementation Notes

The current implementation performs dependency injection manually.

A dedicated dependency injection framework is not used.

This keeps the dependency graph explicit and easy to understand.

Business services never instantiate repositories directly.

Repositories are always injected from the Application Container.

---

## Component Interaction

                        ▼

              Create Repositories

# 7. Task Management Component

## Responsibility

The Task Management component is responsible for managing the complete lifecycle of a task.

A task represents a unit of work performed during a Work Day.

The component ensures that every task is created, validated, managed, and queried according to the business rules defined by the system.

---

## Primary Responsibilities

The component currently provides the following business capabilities:

- Create Task
- Change Task Lifecycle State
- Query Tasks

These operations represent complete business use cases rather than simple database operations.

---

## Inputs

The component receives business commands such as:

- CreateTaskCommand
- TaskLifecycleCommand
- Task Query Requests

Each command contains the information required to perform a complete business operation.

---

## Internal Processing

### Task Creation

Task creation follows the sequence below.

```
Receive CreateTaskCommand
            │
            ▼
Validate Task Category
            │
            ▼
Validate Sub Category
            │
            ▼
Validate Category Relationship
            │
            ▼
Load Work Day
            │
            ▼
Build Task Object
            │
            ▼
Persist Task
            │
            ▼
Create Business Event
            │
            ▼
Return Task
```

---

## Business Validation

Before a task can be created, several validations are performed.

### Category Validation

The selected category must exist.

If the category cannot be found, task creation fails.

---

### Sub Category Validation

The selected sub category must exist.

If it cannot be found, task creation fails.

---

### Classification Validation

The selected sub category must belong to the selected category.

Cross-category combinations are rejected.

---

### Work Day Validation

Every task must belong to an existing Work Day.

Tasks cannot exist independently.

---

## Context Inheritance

One of the most important business rules implemented by the current system is context inheritance.

When optional business information is omitted during task creation, the task automatically inherits the corresponding values from its parent Work Day.

Inherited values include:

- Area
- Site
- Project
- Contractor

This guarantees that every task always contains complete business context.

---

## Lifecycle Management

The component is responsible for managing task state transitions.

Examples include:

- Active
- Completed

The transition rules are centralized inside the Task Lifecycle Service.

Invalid transitions are rejected before data persistence.

---

## Event Generation

Business events are generated automatically after successful business operations.

For example:

Task Creation

↓

Generate Event

↓

Persist Event

Current events include:

- task-started

Future events can be added without modifying existing consumers.

---

## Outputs

The component produces:

- Task
- EventLog

Query operations produce:

- Task Details
- Task Lists
- Task Summaries

---

## Dependencies

The Task Management component depends on:

- Task Repository
- Work Day Repository
- Event Repository
- Task Category Repository
- Sub Category Repository

It does not communicate directly with infrastructure.

---

## Business Rules

Current business rules include:

- Every Task belongs to exactly one Work Day.
- Every Task must have a valid Category.
- Every Task must have a valid Sub Category.
- Sub Category must belong to the selected Category.
- Context may be inherited from the Work Day.
- Every successful task creation generates a business event.

---

## Interactions

The component collaborates with several other business components.

```
                 Task Management
                        │
        ┌───────────────┼────────────────┐
        ▼               ▼                ▼
 Work Day        Classification      Event Log
        │               │                │
        └───────────────┼────────────────┘
                        ▼
                  Repository Layer
```

---

## Architectural Notes

The Task component contains business rules only.

It never performs infrastructure operations directly.

Persistence is delegated to repository interfaces.

This allows the business logic to remain independent from storage implementation.

The current implementation demonstrates a clear separation between:

- Business validation
- Object construction
- Data persistence
- Event generation

This separation makes the component easy to maintain, extend, and test.

# 8. Work Day Management Component

## Responsibility

The Work Day Management component is responsible for managing the lifecycle of a work day.

A Work Day represents the highest business context within the system.

Every operational activity performed by the system begins inside a Work Day.

Tasks, production records and business events are all associated with a Work Day either directly or indirectly.

---

## Primary Responsibilities

The component currently provides the following capabilities:

- Create Work Day
- Manage Work Day Lifecycle
- Query Work Days

The Work Day serves as the business container for daily operations.

---

## Inputs

The component receives business commands including:

- CreateWorkDayCommand
- WorkDayLifecycleCommand
- WorkDay Query Requests

---

## Internal Processing

### Work Day Creation

The creation process follows the sequence below.

```
Receive CreateWorkDayCommand
            │
            ▼
Validate Business Data
            │
            ▼
Build Work Day Object
            │
            ▼
Persist Work Day
            │
            ▼
Generate Business Event
            │
            ▼
Return Work Day
```

---

## Business Context

A Work Day stores the business context shared by all tasks created during that day.

Typical context information includes:

- Area
- Site
- Project
- Contractor

This context becomes the default source of business information for child tasks.

---

## Relationship with Tasks

The relationship between Work Days and Tasks is hierarchical.

```
Work Day
    │
    ├──── Task
    ├──── Task
    ├──── Task
    └──── Task
```

A task cannot exist without a parent Work Day.

The Work Day therefore acts as the root business aggregate for execution activities.

---

## Lifecycle Management

The component manages the business state of a Work Day.

Lifecycle operations are isolated inside the Work Day Lifecycle Service.

Business validation occurs before any state transition.

---

## Outputs

The component produces:

- WorkDay
- WorkDay Details
- WorkDay Summary
- EventLog

---

## Dependencies

The component depends on:

- Work Day Repository
- Event Repository

Query services may also depend on Task Repository to calculate work day summaries.

---

## Business Rules

The current implementation enforces the following rules:

- Every Work Day has a unique identifier.
- A Work Day represents a single business context.
- Tasks inherit context from their Work Day.
- Business events are generated after successful lifecycle operations.

---

## Interactions

```
              Work Day Management
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
      Tasks      Event Logging   Repository
```

---

## Architectural Notes

The Work Day component acts as the root business context of the application.

Rather than duplicating business information across every task, shared information is maintained at the Work Day level and reused by child entities.

This design reduces redundancy while ensuring business consistency.

# 9. Production Tracking Component

## Responsibility

The Production Tracking component is responsible for recording the actual production generated while executing tasks.

Unlike the Task component, which represents planned or active work, this component records measurable business output.

Each production record represents a business fact describing what has been produced during task execution.

---

## Primary Responsibilities

The component currently provides the following capabilities:

- Create Production Record
- Query Production Records

Future capabilities may include:

- Production editing
- Production approval
- Production reconciliation

---

## Inputs

The component receives business commands including:

- CreateProductionRecordCommand
- Production Record Query Requests

A production record always references an existing task.

---

## Internal Processing

### Production Record Creation

The current implementation follows the sequence below.

```
Receive Production Command
            │
            ▼
Load Related Task
            │
            ▼
Validate Task
            │
            ▼
Build Production Record
            │
            ▼
Persist Production Record
            │
            ▼
Return Production Record
```

---

## Relationship with Tasks

Production is always associated with task execution.

```
Task
 │
 ├──── Production Record
 ├──── Production Record
 ├──── Production Record
 └──── Production Record
```

The task provides the execution context for production activities.

---

## Business Rules

The current implementation enforces the following rules.

### Existing Task

A production record cannot be created unless the referenced task exists.

---

### Immutable Business Fact

A production record represents work that has already happened.

Once recorded, it becomes part of the operational history of the system.

---

### Independent Records

Multiple production records may belong to the same task.

Each record represents a separate production event.

---

## Outputs

The component produces:

- Production Record
- Production Record Details
- Production Record Lists

---

## Dependencies

The component depends on:

- Task Repository
- Production Record Repository

It does not communicate directly with infrastructure components.

---

## Interactions

```
          Production Tracking
                    │
         ┌──────────┴──────────┐
         ▼                     ▼
     Task Component      Repository Layer
```

---

## Architectural Notes

Production Tracking is intentionally separated from Task Management.

A task describes the execution of work.

A production record describes the measurable outcome of that work.

Separating these concepts allows the system to support multiple production entries for a single task while preserving a clear business model.

This separation also prepares the system for future reporting, analytics, productivity measurements, and business intelligence without modifying the task lifecycle itself.

# 10. Classification Management Component

## Responsibility

The Classification Management component is responsible for organizing and validating the business classification structure used throughout the system.

Rather than allowing arbitrary task categorization, the system enforces a controlled classification hierarchy to ensure consistency across all business operations.

The component guarantees that every task belongs to a valid business classification.

---

## Primary Responsibilities

The component currently provides the following capabilities:

- Manage Task Categories
- Manage Sub Categories
- Validate Category Relationships
- Provide Classification Queries

---

## Classification Structure

The current implementation follows a hierarchical classification model.

```
Category
    │
    ├──── Sub Category
    ├──── Sub Category
    ├──── Sub Category
    └──── Sub Category
```

Each Sub Category belongs to exactly one Category.

---

## Inputs

The component receives requests from business services requiring classification information.

Typical requests include:

- Find Category
- Find Sub Category
- Validate Classification
- Query Available Categories

---

## Internal Processing

When another component requests classification validation, the process follows the sequence below.

```
Receive Classification Request
              │
              ▼
Load Category
              │
              ▼
Load Sub Category
              │
              ▼
Validate Relationship
              │
              ▼
Return Validation Result
```

---

## Business Rules

The current implementation enforces the following rules.

### Existing Category

Every referenced category must exist.

Missing categories immediately terminate the business operation.

---

### Existing Sub Category

Every referenced sub category must exist.

---

### Valid Relationship

A Sub Category must belong to the selected Category.

The following situation is considered invalid.

```
Category A
        │
        └──────── Sub Category from Category B
```

Cross-category classification is rejected before any business data is persisted.

---

## Outputs

The component provides:

- Category
- Sub Category
- Classification Validation Result
- Classification Queries

---

## Dependencies

The component depends on:

- Task Category Repository
- Sub Category Repository

No dependency exists on infrastructure implementations.

---

## Consumers

The primary consumer of this component is:

- Task Management

Future components may also depend on classification services if they require business categorization.

---

## Error Handling

The component currently produces business-specific errors.

Examples include:

- CategoryNotFoundError
- SubCategoryNotFoundError
- InvalidTaskClassificationError

These errors prevent invalid business data from entering the system.

---

## Architectural Notes

Classification is modeled as an independent business capability rather than simple lookup tables.

This decision centralizes classification rules in one location and prevents duplicate validation logic across multiple business services.

As the system evolves, additional classification levels or validation rules can be introduced without changing Task Management.

# Dependency Rules

The current implementation follows these dependency rules.

## Rule 1

Presentation components never access repositories directly.

They communicate only through application services.

---

## Rule 2

Business services never communicate directly with infrastructure.

All persistence is performed through repository interfaces.

---

## Rule 3

Infrastructure never contains business rules.

Its responsibility is limited to technical implementation.

---

## Rule 4

Reference data never controls business workflow.

It only provides contextual information required by business operations.

---

## Rule 5

Classification validation is centralized.

Business services must never duplicate classification rules.

---

## Rule 6

Business events are created only after successful business operations.

Failed operations never generate business events.

---

## Rule 7

Application Composition is the only component allowed to instantiate repositories and business services.

No other component performs dependency construction.
